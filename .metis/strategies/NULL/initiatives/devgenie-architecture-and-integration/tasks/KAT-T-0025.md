## Goal

Produce a short install-paths reference inside the initiative directory documenting the two ways a user installs katana: (a) standalone (run `/katana-init` directly), (b) through dev-genie orchestration (`/dev-genie-init` walks the registry and offers katana). Captures the decision that katana is standalone-first and dev-genie integration is additive, per the initiative's open questions.

## Deliverables

- Create: `/Users/danielcassil/Code/katana/.metis/strategies/NULL/initiatives/devgenie-architecture-and-integration/install-paths.md`

## Architecture & Contracts

The doc must contain these sections:

1. **Standalone path**:
   - User installs katana plugin (e.g. `claude plugin install katana` or local path).
   - User runs `/katana-init`.
   - Katana scaffolds `.katana/` and registers its MCP server directly via `claude mcp add` (or equivalent).
   - dev-genie is not required; not consulted; not invoked.

2. **dev-genie path**:
   - User runs `/dev-genie-init`.
   - dev-genie orchestration walks registry: guardrails → audit → katana.
   - For katana entry, dev-genie calls `/katana-init` (the same command as the standalone path) — dev-genie does no katana-specific scaffolding.
   - Verification reuses the registry's post-setup verification rules.

3. **MCP registration ownership** — answer the initiative's open question: katana registers its own MCP server during `/katana-init`. dev-genie never registers MCP servers on katana's behalf. This keeps the standalone path complete and avoids a divergent code path.

4. **Cross-references** —
   - Registry entry: `/Users/danielcassil/Code/gaurd-rails-boilerplate/dev-genie/skills/orchestration/SKILL.md` § "3. katana"
   - Detection signals: `/Users/danielcassil/Code/gaurd-rails-boilerplate/dev-genie/skills/project-detection/SKILL.md` § "Katana detection"
