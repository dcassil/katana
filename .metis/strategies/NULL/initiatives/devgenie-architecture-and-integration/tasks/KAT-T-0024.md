## Parent Initiative

[KAT-I-0007 dev-genie Architecture Pattern and Integration](../initiative.md)

## Goal

Extend `dev-genie/skills/project-detection/SKILL.md` so that, in addition to suggesting an application architecture, it emits a `katana_installed` boolean and a `suggests_katana` boolean. These let the orchestration walk default-on or default-off the optional katana prompt without changing orchestration logic.

## Deliverables

- Modify: `/Users/danielcassil/Code/gaurd-rails-boilerplate/dev-genie/skills/project-detection/SKILL.md`

## Architecture & Contracts

Add a new section to the project-detection skill titled `## Katana detection` containing:

1. **`katana_installed`** — true if any of:
   - `.katana/` directory exists in the workspace root.
   - `.katana/config.toml` exists.
   - `claude mcp list` (when available) reports a server whose name matches `katana`.

2. **`suggests_katana`** — true if any of:
   - the repo already uses `.metis/` (signals doc-driven planning affinity; metis users are likely katana audience).
   - `AGENTS.md` or `CLAUDE.md` exists at workspace root and references kanban / decomposition / two-pass.
   - the user has explicitly opted in via flag.

3. **Output schema additions** — show the JSON shape the skill returns; add the two new boolean fields next to the existing `project_kind` and `suggested_architecture`.

Example output block to include verbatim:

```json
{
  "project_kind": "greenfield" | "existing",
  "suggested_architecture": "node-api" | "react-next-vercel-webapp" | ...,
  "katana_installed": false,
  "suggests_katana": false
}
```

## Do / Don't

- Don't change the existing detection logic for `project_kind` or `suggested_architecture` — only add fields.
- Don't auto-install katana from project-detection — this skill is heuristic-only, per its own description.
- Don't shell out to anything beyond the read-only checks already documented in the skill (file-exists checks, optional `claude mcp list`).
- Don't modify orchestration in this task — orchestration consumes these new fields but its update is implicit (registry entry from KAT-T-0023 already says "Default to no unless project-detection signals an agent-workflow project").
- Don't introduce a new tool dependency.

## Acceptance

- [ ] Both `katana_installed` and `suggests_katana` are defined with their signal lists.
- [ ] Output JSON example shows both fields.
- [ ] No other dev-genie file is modified.
