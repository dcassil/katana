# Decision Summary

## Persistence layer

Katana will inherit metis's foundational storage model: Markdown files with YAML frontmatter stored in a hierarchical `.metis/` directory structure, mirrored to a SQLite database. Short codes (PREFIX-TYPE-NNNN format) provide stable, human-readable document identifiers with per-type uniqueness enforcement. Synchronization via idempotent `metis sync` scans the filesystem, detects changes via file hashing, and rebuilds the FTS index—essential for agent-driven workflows where code commits markdown and CI/CD pipelines rebuild indexes. Archive, deferred to post-MVP, will leverage the same framework (logical move to `archived/` directory, boolean flag in database). The persistence layer directly supports katana's doc-driven, version-control-friendly workflow and agent integration. Key risks: uncertainty around how two-pass task variants (high-pass, low-pass) map to storage, concurrent agent updates, and file_hash interaction with agent-driven changes.

## Rules layer

Katana inherits metis's forward-only phase machine, which enforces state-machine transitions per document type and blocks backward movement and phase skips. Exit-criteria gating (a boolean flag checked at transition time) directly supports katana's gate engine vision, though implementation details (whether metis actually enforces before transition vs. advisory-only) require clarification. Lint, a robust rules-based validation system with severity levels, custom rule support, and MCP exposure, directly powers katana's document-completeness checks before phase advancement and decomposition. Validate, a low-level format checker for YAML frontmatter and required fields, depends on metis rather than fork to minimize maintenance burden; katana calls `metis validate` as a pre-flight check for document acceptance. Key risks: metis's phases are hard-coded in the Rust binary, so katana must either fork to add new document types (Product Doc, Epic, User Story, Task[high-pass], Task[low-pass]) or wrap and inject custom types at runtime; lint rule composition and custom field support; validate's lack of MCP exposure.

## Query layer

Katana inherits metis's full-text search (FTS5 with Porter stemmer, Unicode61 collation) and list/filter operations (indexed on document_type and phase for O(log n) performance). These provide the foundation for agent-driven document discovery ("find all todo tasks") and human navigation (kanban views, backlog review). Katana depends on metis's status command for priority-tier logic (blocked → todo → active → completed priority ordering) rather than reimplementing, though the relative timestamps ("2 minutes ago") are not machine-parseable for agent loops; workaround is to query the database directly or request metis add an `--format=agent-json` option with ISO 8601 timestamps. Key risks: FTS5 rebuild performance at scale (10k+ documents), phase-name standardization assumption, missing integration between search and list commands, and status's lossy time format.

## Agent-integration layer

Katana inherits metis's template engine, which supports a fallback chain (project-level `.metis/templates/{type}/`, global `~/.config/metis/templates/{type}/`, compiled-in defaults) and Tera variable substitution, allowing custom templates for gate-related acceptance criteria without forking metis core. Configuration presets (streamlined, direct) control document-type enablement, but presets are fixed and cannot be customized; katana will likely need finer-grained per-type flight-level control, so preset deferral to post-MVP allows time to design a proper configuration hierarchy. Agent-rules generation (CLAUDE.md injection, Cursor rule files, Codex equivalents) is absent from metis entirely and will be katana's own responsibility via pluggable platform adapters that wrap MCP tools in platform-specific native mechanisms (Claude skills, Cursor `.mdc` files, etc.). Code-index (tree-sitter symbol extraction + AI semantic summaries, FTS-indexed) is deferred to post-MVP as a well-isolated, optional feature for agents navigating large codebases. Key risks: template-engine extensibility without forking DocumentType enum, preset inflexibility, agent-rules design complexity, and code-index context-window and language-coverage challenges.

## MCP surface

Katana will inherit 13 metis MCP tools spanning CRUD, search, and lifecycle operations. Initialize_project, create_document, read_document, edit_document, list_documents, search_documents, transition_phase, archive_document, and list_patterns are all generic enough to adapt cleanly to katana's document hierarchy. Reassign_parent and lint_document/lint_workspace are deferred to post-MVP pending clarification of katana's backlog model and lint-rule framework. Arch_lint (architecture-pattern validation) is skipped as tangential to document management. Summary: 9 Inherit, 3 Defer, 1 Skip. MCP tools are the canonical interface through which katana agents invoke workflows, making platform-agnostic tool design critical. Key risks: document-type and hierarchy coupling in create/transition; phase-machine finalization before MCP design; exit-criteria gating semantics; backlog-category taxonomy mismatch.

## Katana additions

Katana introduces seven major capabilities beyond metis. Two-pass task model (high-pass scaffold generation, low-pass implementation fill-in, distinct document types and phase sets per variant) is MVP-critical to the vision and success criteria. Pluggable agent-platform adapter (Claude/Cursor/Codex/generic behind a common MCP interface) is an MVP success criterion but requires new architecture (abstract platform-adapter interface with concrete implementations). Pluggable kanban-board adapter (internal default, swappable to Jira/Linear) is post-MVP but design must accommodate it (board operations: list, create, update, delete; one/two-way sync; orphan handling). Pluggable storage backend (sqlite+markdown default, alternatives possible) is post-MVP; persistence layer must abstract backends but MVP locks in sqlite+markdown. Gate engine as composable config (pre-decomposition, exit-criteria, parent-readiness gates defined in markdown+config, not hardcoded) is MVP-critical to the "gates must be authorable by humans without code changes" principle; requires refactoring metis's phase machine to invoke pluggable gate hooks. Work-eval-gate-loop runtime (work → eval → gate → loop → done state machine per task, intelligently dispatches strong/weak models per pass) is MVP-critical to the two-pass execution vision; distinct from metis's passive document system—katana needs active orchestration. New document hierarchy (Product Doc → Epic → User Story → Task[high-pass] → Task[low-pass] + optional Task[UI]) is MVP-critical structural foundation; extends metis's 4-level hierarchy to 7 levels, requires new document types in schema, phase sets, templates, and parent constraints. Open questions span task-variant phase sets, adapter discovery, board sync direction, backend contract, gate rule syntax, loop checkpoint/pause semantics, and terminal document types.

- **From survey-rules.md:** Phase machine hard-coded in binary; fork vs. wrap strategy required. Exit-criteria gating enforcement timing (before or advisory)? Custom phases beyond standard 5–6 supported in config or source modification? Phase tags auto-update or manual sync? Lint rule composition and evaluation order? Custom rule field access? `[lint.phase_gates]` usage details? Document-to-document cross-validation?
- **From survey-query.md:** FTS5 rebuild performance at scale? Query sanitization over-protective for agents? Multilingual stemming? Search/list filtering integration? Phase-name standardization assumption? Status relative timestamps lossy for agents (requires ISO 8601 workaround or metis extension). `blocked_by` field not populated in status. Missing count metrics by phase.
- **From survey-agent-integration.md:** Template DocumentType enum forking required to add new types; is this acceptable? Frontmatter schema validation enum-based or config-extensible? Template variables extensible without code? Presets hard-coded and inflexible; final katana hierarchy may not fit streamlined/direct preset model. Agent-rules generation design (CLAUDE.md vs. Cursor .mdc vs. OpenAI/Codex approach; version-control vs. regenerate strategy). Code-index language coverage (TS/JS/Rust/Python/Go current; C#/Java/others needed?).
- **From survey-mcp.md:** Initialize_project: separate katana implementation justified or mirror metis? Create_document: document-type coupling and backlog-category mismatch. Transition_phase: phase definitions finalize before MCP design (major dependency). Archive_document: cascade behavior assumptions. Reassign_parent: backlog taxonomy mismatch. Lint tools: rule definition framework not yet designed. List_patterns: pattern system adoption undecided.
- **From survey-katana-additions.md:** Two-pass task model: identical phase sets or different per variant? Parent-child or independent relationship? Type-specific validation? Agent-platform adapter: contract definition (MCP wrapper pattern vs. native integration)? Platform discovery/registration? Kanban board: operation set (CRUD), sync direction (one/two-way), orphan handling, cascade mapping. Storage backend: contract (store/retrieve/FTS/short-code/list/archive/sync)? Uniqueness enforcement cross-backend? Migration story? Gate engine: rule syntax (YAML/DSL/Tera)? Cross-document references? Document interaction (suggestions on gate failure)? Task variant sequencing or co-occurrence? 1:1 metis mapping or migration needed?
- **Cross-survey:** Agent-integration layer appears to skip agent-rules generation but MVP success criterion requires katana to support "Same katana workflow on Claude Code, Cursor, Codex"—conflict between skip and requirement; resolved by making agent-rules generation a new subsystem outside metis survey scope. Phase machine hard-coded binary in metis vs. katana's requirement to support new document types (Product Doc, Epic, User Story, Task variants)—major architectural decision pending: fork metis or implement pluggable type registry.
