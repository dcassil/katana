# Survey metis MCP tool surface

## Parent Initiative

[[KAT-I-0008]]

## Goal

Enumerate every MCP tool exposed by `metis mcp` and produce one decision row per tool (Inherit / Depend / Defer / Skip) for Katana's own MCP server. The Katana vision requires an MCP surface as a first-class delivery mechanism, so this survey directly drives the MCP server initiative.

## Deliverables

- `/Users/danielcassil/Code/katana/.metis/strategies/NULL/initiatives/metis-feature-parity-survey/notes/survey-mcp.md`

## Survey criteria

For **each** MCP tool listed below, answer:

1. **Tool name** — exact `mcp__plugin_metis_metis__*` identifier.
2. **What it does** — one-sentence purpose grounded in observed MCP server instructions / metis CLI parity.
3. **Inputs / outputs** — best-effort summary of arguments and return shape. If unknown, say so explicitly.
4. **CLI counterpart** — which `metis <subcommand>` (if any) it mirrors.
5. **Katana need.** MVP / Post-MVP / Never.
6. **Decision.** Inherit (mirror in Katana MCP under a renamed tool) | Depend (call metis's MCP from Katana) | Defer | Skip.
7. **Rationale.** 2–4 sentences. Note especially whether the tool's contract is generic (CRUD, search) or coupled to metis's specific phase machine / doc types.
8. **Katana name suggestion.** Proposed Katana MCP tool name (e.g., `katana_create_document`). Naming only — no implementation.
9. **Risks / open questions.** Bullets.

## Do / Don't

- **Don't** design the Katana MCP server. You only decide which tools to mirror and propose names.
- **Don't** invoke any of the metis MCP tools to mutate state. Read-only inspection (e.g., `list_documents` against your own workspace) is acceptable but not required — CLI help is sufficient evidence.
- **Don't** write code.
- **Don't** speculate on Katana-specific tools (e.g., `decompose`, `run_two_pass`); those belong to the MCP-server initiative, not this survey.
- **Don't** invent inputs/outputs you cannot verify — say "unknown — to investigate" instead.
- **Do** include the summary table at the top so the roll-up task can copy it directly.
- **Do** flag tools whose contract is tightly coupled to metis's doc types (vision/strategy/initiative/task/adr) versus tools that are generic enough to inherit cleanly.
