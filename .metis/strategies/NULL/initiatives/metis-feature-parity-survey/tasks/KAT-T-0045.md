# Catalogue Katana additions beyond metis

## Parent Initiative

[[KAT-I-0008]]

## Goal

Enumerate every Katana capability that has **no metis counterpart** — the things Katana adds that an "inherit metis" approach would not give us for free. This complements the four metis-surveys (KAT-T-0040..0043) and the MCP survey (KAT-T-0044) by ensuring nothing is silently lost. This task is decision-only (no code).

## Deliverables

- `/Users/danielcassil/Code/katana/.metis/strategies/NULL/initiatives/metis-feature-parity-survey/notes/survey-katana-additions.md`

## Survey criteria

The Katana vision identifies these distinct additions. For **each** addition below, answer in order:

1. **What is it?** One-paragraph definition grounded in the Katana vision file.
2. **Why is it not in metis?** One or two sentences explaining the gap (e.g., metis is single-pass, single-platform, single-storage).
3. **MVP-required?** Yes / No, with a vision-citation phrase.
4. **Touches which metis-survey row(s)?** List the surveyed metis capability rows (from KAT-T-0040..0044) this addition modifies, replaces, or extends. If none, say "greenfield — no metis row touched."
5. **Inheritance impact.** If Katana inherits the corresponding metis layer, what must change to accommodate this addition? Be specific (e.g., "phase machine must accept new doc types Task[high-pass] / Task[low-pass]").
6. **Open questions.** Bullets.

### Additions in scope (from the initiative file)

- **A1: Two-pass task model** — Task[high-pass] (scaffold) and Task[low-pass] (fill-in) as distinct doc types with different model bindings.
- **A2: Pluggable agent-platform adapter** — Claude / Cursor / Codex behind a common interface (slash commands, MCP, agent-rules files).
- **A3: Pluggable kanban-board adapter** — internal default + interface for Jira/Linear.
- **A4: Pluggable storage backend** — sqlite+md default, contract for alternatives.
- **A5: Gate engine as first-class composable config** — pre-decomposition gates, phase-transition exit gates, parent-readiness gates, authorable as config.
- **A6: Work-eval-gate-loop runtime per task** — workflow loop driving a doc through phases with the right model per step.
- **A7: New document hierarchy** — Product Doc → Epic → User Story → Task[high-pass] → Task[low-pass] → optional Task[UI].

## Do / Don't

- **Don't** add additions not present in the vision file. If you think one is missing, list it under a final "## Candidate additions (not in vision)" section with a note that the roll-up task should validate it.
- **Don't** write code or design any of these additions.
- **Don't** decide Inherit/Depend/Defer/Skip on these — that vocabulary is for metis capabilities. These are *additions*; the question is only "MVP-required or not?" and "what does it impact?"
- **Don't** modify the initiative or vision files.
- **Do** quote vision phrases verbatim when citing MVP-required.
- **Do** be specific in "Inheritance impact" — name the metis subsystem affected.
