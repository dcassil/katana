# Survey metis rules layer (phase machine, lint, validate)

## Parent Initiative

[[KAT-I-0008]]

## Goal

Survey the three rules-layer capabilities of metis — the phase state machine (forward-only transitions per document type), `metis lint` / `lint_workspace`, and `metis validate` — and produce one decision row per capability with rationale grounded in Katana's two-pass task model and gate engine.

## Deliverables

- `/Users/danielcassil/Code/katana/.metis/strategies/NULL/initiatives/metis-feature-parity-survey/notes/survey-rules.md`

## Survey criteria

For **each** capability below, answer all of these questions in order:

1. **What does metis provide?** Behavior description grounded in CLI help and observed transitions.
2. **How is it implemented?** Surface (CLI subcommand, MCP tool, frontmatter tag like `#phase/active`, severity flag). Cite exact names.
3. **Evidence sources.** Paths and command outputs you inspected.
4. **Katana need.** MVP / Post-MVP / Never, with explicit reference to the vision (gates, phase progression, two-pass tasks).
5. **Decision.** Inherit | Depend | Defer | Skip.
6. **Rationale.** 2–4 sentences. Note especially: does Katana need to *extend* the phase machine for new doc types (Product Doc, Epic, User Story, Task[high-pass], Task[low-pass])?
7. **If Inherit:** which metis source files / modules to copy or fork.
8. **Risks / open questions.** Bullets.

### Capabilities in scope

- **C1: Phase machine** — forward-only transitions, per-doc-type phase sets, `metis transition <SHORT_CODE> [PHASE]`, `#phase/<name>` tags, exit-criteria gating.
- **C2: Lint** — `metis lint [SHORT_CODE]`, severity levels (error/warning/info), workspace-wide lint, MCP `lint_document` / `lint_workspace`.
- **C3: Validate** — `metis validate <FILE_PATH>` — frontmatter / structural validation of a single document file.
