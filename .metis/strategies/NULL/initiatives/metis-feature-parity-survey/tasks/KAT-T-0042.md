# Survey metis query layer (search, list, status)

## Parent Initiative

[[KAT-I-0008]]

## Goal

Survey the three query-layer capabilities of metis — full-text search (FTS5), `metis list` filtering, and `metis status` actionable-items roll-up — and produce one decision row per capability with rationale grounded in Katana's agent-driven workflow loop.

## Deliverables

- `/Users/danielcassil/Code/katana/.metis/strategies/NULL/initiatives/metis-feature-parity-survey/notes/survey-query.md`

## Survey criteria

For **each** capability below, answer all of these questions in order:

1. **What does metis provide?** Behavior description grounded in CLI help and the sqlite schema (`document_search` FTS5 virtual table).
2. **How is it implemented?** Surface (CLI subcommand, MCP tool name, sqlite virtual table / index). Cite exact names.
3. **Evidence sources.** Paths, commands, sqlite queries.
4. **Katana need.** MVP / Post-MVP / Never. Note: agent loops need cheap query-by-status; humans need search.
5. **Decision.** Inherit | Depend | Defer | Skip.
6. **Rationale.** 2–4 sentences.
7. **If Inherit:** which metis source / migrations / FTS config to copy.
8. **Risks / open questions.** Bullets.

### Capabilities in scope

- **C1: Search** — `metis search <QUERY>`, output formats (table/compact/json), the `document_search` FTS5 virtual table with porter+unicode61 tokenizer, MCP `search_documents` tool.
- **C2: List** — `metis list` with `--document-type`, `--phase`, `--all`, `--include-archived` filters and JSON output, MCP `list_documents`.
- **C3: Status** — `metis status` "actionable items" view, JSON output, `--include-archived`.
