## Architecture & Contracts

### Source-of-truth: passing fixtures

The five `pass/*.md` files MUST be a self-consistent tree under prefix `KAT-` and numbering `8001`:

```
KAT-PD-8001 (product-doc, system-design, phase: published)
└── KAT-E-8001 (epic, major-feature, phase: ready)
    └── KAT-US-8001 (user-story, interface-contract, phase: ready)
        ├── KAT-TH-8001 (task-high-pass, phase: todo)
        └── KAT-TL-8001 (task-low-pass, phase: todo, scaffold_task: KAT-TH-8001)
```

Use the schema in `/Users/danielcassil/Code/katana/docs/schema/frontmatter.md` and templates under `/Users/danielcassil/Code/katana/templates/` byte-for-byte for headings. Fill every REQUIRED section with at least one sentence of real content (no template instructional `_..._`, no `{{var}}`, no bare `REQUIRED`/`OPTIONAL` markers). The product-doc must list `KAT-E-8001` under `## Child Epics`; the epic must list `KAT-US-8001` under `## Child User Stories`; the user-story must list `KAT-TH-8001` under `### High-pass` and `KAT-TL-8001` under `### Low-pass` inside `## Child Tasks`. Each doc must include `## Exit Criteria` (or per-level equivalent) with at least one `- [ ]` checklist item; `exit_criteria_met` MUST be `false` on each.

### Failing fixture format

Each `fail/*.md` is a single document. Wherever possible, model it after the corresponding `pass/*.md` and mutate the **one** field/section that triggers the target reason code. Add a top-of-file HTML comment naming the expected reason code and gate:

```markdown
<!-- expected-gate: frontmatter-schema -->
<!-- expected-code: frontmatter.invalid-subtype-for-level -->
```

The test runner (KAT-T-0072) reads these comments to know what to assert.

## Do / Don't

- DO place the two HTML comments at the **very top** of every failing fixture (line 1 and line 2). The test runner expects this.
- DO use prefix `KAT-` and the numeric range `8001–8099` for fixtures so they don't collide with worked-example `9001+`.
- DO ensure each failing fixture triggers **only** the named reason code under the named gate; other gates may pass or emit incidental warnings, but the named gate must produce the named code.
- DON'T add code, scripts, or test runners to this directory; this task is fixtures-only.
- DON'T copy the worked-example tree byte-for-byte — renumber to `8xxx` and adjust cross-refs.
- DON'T introduce any `{{var}}` placeholders in `pass/` fixtures.
- DON'T leave a fixture without its expected-gate / expected-code comment header.
