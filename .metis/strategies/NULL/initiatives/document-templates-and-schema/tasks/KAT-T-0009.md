## Goal

Produce filled-in worked-example documents — one per level — that form a complete, self-consistent tree (Product Doc → Epic → User Story → Task[high-pass] + Task[low-pass]). These fixtures will be consumed by KAT-I-0002 (validation gates) as test inputs and by humans as reference exemplars.

## Deliverables

- `fixtures/worked-example/product-doc.md` (new) — short_code `KAT-PD-9001`
- `fixtures/worked-example/epic.md` (new) — short_code `KAT-E-9001`, parent `KAT-PD-9001`
- `fixtures/worked-example/user-story.md` (new) — short_code `KAT-US-9001`, parent `KAT-E-9001`
- `fixtures/worked-example/task-high-pass.md` (new) — short_code `KAT-TH-9001`, parent `KAT-US-9001`
- `fixtures/worked-example/task-low-pass.md` (new) — short_code `KAT-TL-9001`, parent `KAT-US-9001`, `scaffold_task: KAT-TH-9001`
- `fixtures/worked-example/README.md` (new) — index linking the five docs and stating their relationships

## Architecture & Contracts

### Domain for the example

Use a small, generic domain so the example is portable: a CLI that reads a JSON config and prints a formatted report. Concretely:

- Product Doc subtype: `system-design`. Describes the CLI as a system with input/output boundaries.
- Epic subtype: `major-feature`. Adds a `--filter` flag to the CLI.
- User Story subtype: `interface-contract`. Defines `applyFilter(records, filter): Record[]` signature.
- Task[high-pass]: scaffolds `src/filter.ts` with the `applyFilter` export and placeholder body throwing `TODO`.
- Task[low-pass]: implements `applyFilter` body against the scaffold.

### Rules

- All five docs MUST satisfy the templates produced in KAT-T-0003 through KAT-T-0007 — every REQUIRED section present, CONDITIONAL sections matching subtype, no extra sections.
- Frontmatter MUST validate against KAT-T-0001 (use the `9001` numeric block to keep fixtures separate from real docs).
- Cross-references MUST resolve within the fixture set (parents, scaffold_task).
- The high-pass and low-pass tasks MUST agree on file path (`src/filter.ts`), exported symbol (`applyFilter`), and signature.

### `README.md`

Sections:
1. `# Worked Example: Filter Flag`
2. `## Tree` — ASCII tree of the five short codes and their relationships.
3. `## Files` — bulleted list with one-line purpose per fixture.
4. `## Use` — one paragraph stating these are fixtures for KAT-I-0002 tests; do not edit without updating the validation suite.

## Do / Don't

- DO make the five docs consistent — same domain, same file paths, same types across high-pass and low-pass.
- DO use the `9001` numeric tail to keep fixtures out of real-doc range.
- DON'T invent new template sections in fixtures; if a section is missing from a template, fix the template task instead.
- DON'T put real implementation code in the high-pass fixture's scaffold bodies — use the `throw new Error("TODO: implement (KAT-TL-9001)")` convention.
- DON'T reference content that hasn't been authored (no dangling short codes).
