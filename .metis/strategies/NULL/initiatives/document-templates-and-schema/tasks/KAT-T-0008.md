## Parent Initiative

[KAT-I-0001 — Document Templates and Schema](../initiative.md)

## Goal

Author the optional Task[UI] template — used when a user story's UI work doesn't cleanly split into scaffold/fill-in passes (e.g. visual polish, single-component build). One pass, UI-tier model.

## Deliverables

- `templates/task-ui/_base.md` (new)

## Architecture & Contracts

Frontmatter: per KAT-T-0001 task-ui field set. REQUIRED: `parent: KAT-US-####`, `model_tier: ui`. NO `pass` field, NO `scaffold_task` field.

Body sections (in order, all REQUIRED unless marked OPTIONAL):

1. `## Parent User Story` — one-line link.
2. `## Goal` — one paragraph: which UI surface or component is built.
3. `## Files to Create or Modify` — bulleted list with one-line purpose each.
4. `## Visual / Interaction Spec` — references to design source (Figma link, screenshots) or inline description. REQUIRED.
5. `## States & Edge Cases` — bulleted list (loading, empty, error, etc.).
6. `## Allowed Scope` — REQUIRED. What the executor may add (styles, components within scope) and may NOT add (new routes, data layer changes, new dependencies).
7. `## Acceptance` — checklist: visual matches spec, all states implemented, no out-of-scope changes.
8. `## Cross-References` — short codes only.

Italic one-line hint under each heading.

## Do / Don't

- DO keep the template single-pass — no scaffold/fill split.
- DO require an explicit allowed/denied scope (Section 6).
- DON'T add `pass` or `scaffold_task` fields to the frontmatter spec.
- DON'T duplicate user-story UI content; link by short code.

## Acceptance

- [ ] File exists at `templates/task-ui/_base.md`.
- [ ] All 8 sections present in order with italic hints.
- [ ] Frontmatter omits `pass` and `scaffold_task`; declares `model_tier: ui`.
- [ ] `## Allowed Scope` lists both allowed and denied additions.
