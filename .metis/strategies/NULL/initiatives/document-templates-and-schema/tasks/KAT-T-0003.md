## Parent Initiative

[KAT-I-0001 — Document Templates and Schema](../initiative.md)

## Goal

Author the Product Doc template family. One base template plus three subtype overlays (`architecture`, `system-design`, `ui`) — each overlay is a short markdown file listing which sections become REQUIRED for that subtype. The base template covers the union of sections; overlays select.

## Deliverables

- `templates/product-doc/_base.md` (new)
- `templates/product-doc/architecture.md` (new)
- `templates/product-doc/system-design.md` (new)
- `templates/product-doc/ui.md` (new)

## Architecture & Contracts

### `_base.md`

Frontmatter: per KAT-T-0001 product-doc field set. Use `{{placeholders}}` for runtime fields.

Body sections (in this order; mark each REQUIRED / CONDITIONAL):

1. `## Purpose` — REQUIRED.
2. `## Audience` — REQUIRED.
3. `## Problem & Current State` — REQUIRED.
4. `## Goals / Non-Goals` — REQUIRED.
5. `## Architecture Overview` — CONDITIONAL (subtype = architecture | system-design).
6. `## System Components & Boundaries` — CONDITIONAL (subtype = system-design).
7. `## Data Model Sketch` — CONDITIONAL (subtype = system-design).
8. `## UX Surfaces & Flows` — CONDITIONAL (subtype = ui).
9. `## Visual / Interaction Principles` — CONDITIONAL (subtype = ui).
10. `## Constraints` — REQUIRED.
11. `## Success Criteria` — REQUIRED.
12. `## Open Questions` — OPTIONAL.
13. `## Child Epics` — REQUIRED (list of epic short codes; may be empty before decomposition).

Each section heading must be followed by a one-line italic hint, e.g. `_What problem does this solve and for whom?_`.

### Subtype overlay files

Each overlay file has this structure only:

```
# Product Doc — <subtype>

## Required Sections
- ## Purpose
- ## Audience
- ...

## Conditional Sections (made REQUIRED for this subtype)
- ## Architecture Overview
- ...

## Forbidden Sections
- (sections from _base that should NOT appear for this subtype, if any)
```

Overlays do NOT duplicate the base template body. They reference section headings only.

#### Required-section sets per subtype

- `architecture.md` — adds `## Architecture Overview`. UX sections forbidden.
- `system-design.md` — adds `## Architecture Overview`, `## System Components & Boundaries`, `## Data Model Sketch`. UX sections forbidden.
- `ui.md` — adds `## UX Surfaces & Flows`, `## Visual / Interaction Principles`. Architecture/data-model sections forbidden.

## Do / Don't

- DO use frontmatter exactly as specified in KAT-T-0001 (don't redefine fields).
- DO use short codes from KAT-T-0002 in placeholders.
- DON'T include implementation guidance, JSDoc, or TypeScript.
- DON'T duplicate sections across overlay files; overlays select from base.
- DON'T add prose describing how validation works — that's KAT-I-0002.
- DON'T put example content in the base template; that belongs in fixtures (KAT-T-0009).
