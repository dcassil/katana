## Parent Initiative

[KAT-I-0001 — Document Templates and Schema](../initiative.md)

## Goal

Produce the canonical frontmatter schema spec that every Katana document template will conform to. This is a single markdown reference doc enumerating every frontmatter field, its type, whether it is REQUIRED or CONDITIONAL (and on what), and the doc types that use it. All other tasks in this initiative reference this spec rather than redefining fields.

## Deliverables

- `docs/schema/frontmatter.md` (new)

## Architecture & Contracts

`docs/schema/frontmatter.md` must contain the following sections, in order:

1. `# Katana Frontmatter Schema`
2. `## Overview` — one paragraph: YAML frontmatter, kebab-case ids, short codes per KAT-T-0002.
3. `## Common Fields` — table with columns: `field | type | required | applies_to | description`.
4. `## Per-Type Field Sets` — one subsection per doc type listing extra fields.
5. `## Validation Rules` — bulleted enumeration of cross-field rules.
6. `## Examples` — minimal valid frontmatter blocks for each doc type (no body).

### Common fields (REQUIRED on all doc types)

```yaml
id: kebab-case-string         # unique within workspace
level: product-doc | epic | user-story | task-high-pass | task-low-pass | task-ui
title: "Human readable"
short_code: "KAT-XX-NNNN"     # see KAT-T-0002
subtype: architecture | system-design | ui | major-feature | interface-contract | null
created_at: ISO-8601 datetime
updated_at: ISO-8601 datetime
archived: boolean
tags: [string]                # must include "#<level>" and "#phase/<phase>"
exit_criteria_met: boolean
phase: todo | active | completed | ...   # valid set per level
```

### CONDITIONAL fields

- `parent: <short_code>` — REQUIRED for epic, user-story, task-*; ABSENT for product-doc.
- `blocked_by: [short_code]` — OPTIONAL; default `[]`.
- `pass: high | low` — REQUIRED on task-high-pass / task-low-pass; ABSENT on others. (Redundant with `level` but kept for query convenience — call this out.)
- `model_tier: strong | cheap | ui` — REQUIRED on task-*; advisory on others.
- `scaffold_task: <short_code>` — REQUIRED on task-low-pass (points at the high-pass task whose scaffold it fills); ABSENT elsewhere.
- `story_id: <short_code>` — REQUIRED on task-*; mirrors `parent` when parent is a user-story.

### Validation rules to list

- `subtype` enum is constrained per `level` (product-doc accepts `architecture | system-design | ui`; epic accepts `architecture | major-feature | ui`; user-story accepts `architecture | interface-contract | ui`; tasks accept `null`).
- `tags` MUST include `#phase/<phase>` matching the `phase` field.
- `short_code` MUST match the regex defined in KAT-T-0002.
- `parent.level` MUST be exactly one tier above `level` in the hierarchy.
- `task-low-pass.scaffold_task` MUST resolve to a `task-high-pass` whose `parent` equals this doc's `parent`.

## Do / Don't

- DO write this as a human-readable spec, not a JSON Schema or Zod file. No code beyond YAML/markdown examples.
- DO state required vs. conditional explicitly per field.
- DON'T invent fields not listed above.
- DON'T specify validation engine behavior, error messages, or how rules are enforced — that belongs to KAT-I-0002.
- DON'T duplicate the short-code regex; reference KAT-T-0002 by short code.
- DON'T add JSDoc or TypeScript types; this is a markdown-only deliverable.
