# CLI: katana install <platform>

## Parent Initiative
KAT-I-0006 — Agent Platform Strategy Pattern.

## Goal
Add a CLI entry point `katana install <platform>` that resolves the adapter via the registry, runs `install()`, and prints a human-readable report. This is the user-facing surface invoked by `/katana-init` (per `install-paths.md`).

## Deliverables
- `src/cli/install.ts` exporting `runInstall(argv: string[]): Promise<number>` (exit code).
- Wired into `src/cli/index.ts` via subcommand dispatch (create or extend that file).
- `tests/cli/install.test.ts`

## Architecture & Contracts
Usage:
```
katana install <platform> [--workspace <path>] [--katana-root <path>] [--mcp-command <cmd>] [--mcp-args <a,b>] [--dry-run] [--force]
```
Defaults:
- `--workspace` = `process.cwd()`
- `--katana-root` = `<workspace>/.katana`
- `--mcp-command` = `npx`
- `--mcp-args` = `-y,katana-mcp`

Behavior:
1. Parse argv (use a minimal parser; no new deps if avoidable — `node:util` `parseArgs` is acceptable).
2. `getAdapter(platform).install({...opts})`.
3. Print a table of `WrittenFile` entries (path, action, bytes), then summary line `installed <platform>: N created, M updated, K skipped, mcp=<bool>`.
4. Exit 0 on success, 1 on error, 2 on unknown platform.

`--dry-run` MUST NOT write any file but MUST emit the same report (with `action` reflecting what would happen).

## Do / Don't
- DO list available platforms when `<platform>` is missing or unknown.
- DO surface adapter `warnings` to stderr.
- DON'T register the MCP server via `claude mcp add`; the adapter writes config files directly.
- DON'T require a previously initialized `.katana/` — install can run before/independently of `/katana-init` document scaffolding.

## Acceptance
- `katana install claude-code --workspace /tmp/x` exits 0 and matches golden report.
- `katana install bogus` exits 2 with helpful message.
- `--dry-run` produces no filesystem changes.
