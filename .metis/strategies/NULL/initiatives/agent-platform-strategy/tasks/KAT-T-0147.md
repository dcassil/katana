# Adapter Registry

## Parent Initiative
KAT-I-0006 — Agent Platform Strategy Pattern.

## Goal
Provide a single lookup point that maps `PlatformId` to a factory returning a fresh `PlatformAdapter` instance. Used by the install CLI and tests.

## Deliverables
- `src/platform/registry.ts`
- `tests/platform/registry.test.ts`

## Architecture & Contracts
```ts
import type { PlatformAdapter, PlatformId } from "./port";

export type AdapterFactory = () => PlatformAdapter;

const REGISTRY: Record<PlatformId, AdapterFactory> = {
  "claude-code":  () => new ClaudeCodeAdapter(),
  "cursor":       () => new CursorAdapter(),
  "openai-codex": () => new OpenAICodexAdapter(),
};

export function getAdapter(id: PlatformId): PlatformAdapter { /* throws on unknown */ }
export function listPlatforms(): PlatformId[];
export function registerAdapter(id: PlatformId, factory: AdapterFactory): void; // for tests/extensions
```
Unknown id throws `UnknownPlatformError` with message `"Unknown platform: <id>. Known: claude-code, cursor, openai-codex"`. Registration is module-init time; `registerAdapter` exists for test injection only.

## Do / Don't
- DO use dynamic imports (`await import("./claude-code")`) only if circular-import problems arise; otherwise plain imports.
- DON'T hold adapter state at module scope — factory returns a fresh instance per call.
- DON'T expose the underlying `REGISTRY` map.

## Acceptance
- `getAdapter("claude-code").id === "claude-code"`.
- `getAdapter("nope")` throws `UnknownPlatformError`.
- `listPlatforms()` returns all three ids in stable insertion order.
- `registerAdapter` lets a test stub override an existing entry and restore.
